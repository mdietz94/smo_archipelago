# Architecture

Three tiers, each independently restartable:

```
[ Switch / SMO ]                [ PC Bridge (Python) ]            [ AP server ]
  exlaunch subsdk9                 asyncio                           archipelago.gg
  LunaKit headers                  CommonContext (Archipelago)       or self-host
  ImGui overlay (M8)   <--TCP-->   Switch TCP server  <--websocket-->
  Hud overlay (M3)                 Web tracker (Flask, M5)
                                   Forked apworld (vendored data)
```

## Why a PC bridge

The Archipelago wire format is a websocket with per-message-deflate compression and TLS. Implementing all three on Switch (inside an `exlaunch` `subsdk9`) is a significant undertaking with poor return: months of work, fragile, hard to update. A Python bridge derived from `Archipelago/CommonClient.py` solves it in a few hundred lines and has the additional benefits:

- The bridge can be updated independently of the Switch module (no re-flashing).
- The bridge hosts the web tracker — no on-Switch UI work for tracker visualization.
- The bridge can run alongside other AP clients on the same machine.
- Reconnect / deflate / TLS bugs are caught upstream by the AP team.

The Switch only needs to speak a small line-delimited JSON protocol over a single TCP socket on the LAN — see [wire-protocol.md](wire-protocol.md).

## Module identity

The forked apworld registers as `Manual_SMO_archipelago` (vs. the Manual client's `Manual_SMO_mp3`). The bridge's `Connect` packet uses this exact game name. Seeds generated by either world are not interchangeable; that's intentional — the real-client seeds gain richer enforcement options the Manual client cannot honor.

## Process boundaries

| Process | Owns |
|---|---|
| Switch module (`subsdk9`) | hooks, game-state mirror, capture lock enforcement, HUD overlay |
| Bridge (`smo-ap-bridge`) | AP websocket, datapackage, web tracker, replay-on-reconnect |
| Browser tracker (web UI) | view-only (polls `/api/snapshot`) |

## Threading

| Thread | Inside | Owns |
|---|---|---|
| SMO main (frame thread) | hooks, drawMain trampoline | `ApState::applyOnFrame()`, HUD draw, hook callbacks |
| AP socket thread | `ApClient::loop` | `nn::socket` recv/send, JSON parse |
| Bridge asyncio loop | bridge | AP websocket, Switch TCP server |
| Bridge web thread | bridge | Flask serving the tracker UI |

The Switch's two threads coordinate through SPSC ring buffers (`outbound_checks`, `inbound`, `outbound_status`) plus `std::atomic`s — no mutexes.

## Key data flows

### Player collects a moon

```
SMO frame → MoonGetHook trampoline
         → reportMoonChecked(kingdom, shine_id)
         → enqueue Check on outbound_checks
AP socket thread → drains outbound_checks → wire {"t":"check",...}
Bridge asyncio   → on_check → CommonContext.send_msgs(LocationChecks)
AP server        → broadcasts ReceivedItems to recipient
```

### Player receives a moon item from another world

```
AP server        → ReceivedItems
Bridge asyncio   → classify via apworld data → wire {"t":"item",...}
AP socket thread → push onto inbound ring
SMO frame thread → ApState::applyOnFrame → MoonApply::grantShine
                 → GameDataHolder::setShineGet → moon flags update; gates open
```

### Player throws Cappy at a locked enemy

```
SMO frame → CaptureStartHook (HOOK_DEFINE_REPLACE if locked path)
         → CaptureGate::captureBlocked(name) ?
         → playSE_NG(); return false  (no possession)
```

## Responsibilities map

| Concern | Switch module | Bridge |
|---|---|---|
| AP websocket / deflate / TLS | — | yes |
| Datapackage / id resolution | — | yes |
| Item classification | — | yes (Moon/Capture/Kingdom/Shop/Other) |
| Moon flag writes | yes | — |
| Capture lock enforcement | yes | — |
| Goal detection | yes (hook) | yes (forwards to AP) |
| Replay on reconnect | applies idempotently | sends replay |
| Web tracker | — | yes (M5) |
| In-game tracker | yes (HUD M3, ImGui M8) | — |

## Why we link LunaKit but don't fork it

LunaKit is the closest mod to ours in spirit (also `subsdk9`, also exlaunch). We need its `tryGetGameDataHolder`, `tryGetPlayerActorHakoniwa`, `tryGetStageScene`, `warpPlayer`, the `DevGuiWindow` base for M8, and its working `nn::socket` bring-up sequence. Forking LunaKit means tracking its rapid churn forever; linking against it as a submodule lets us update at our own pace.

Coexistence: only one `subsdk9` can be installed at a time. M8 ships a combined LunaKit+AP build for users who want both. Until then, users pick one.
